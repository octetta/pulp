background

            
Small machines. Immediate sound.

          

          
        

        

          This is the short personal context slide. Use it to explain the path into this project: old sound hardware, simple command interfaces, handmade tools, and the long-running urge to make sound systems that feel direct enough to poke at in real time. The older slide title was: I keep coming back to small machines that make sound feel immediate.
        

      


      

        
the itch

        
I wanted command-line immediacy for sound.

        

          For years, I wanted a simple way to play with sound without booting a giant tool or wiring together a whole environment first. This is influenced by old PC sound hardware, 80s analog and digital synthesizers, command prompts, tiny notations, and the pleasant danger of making something immediate enough to surprise me.

the stack

        
Text -> engine -> speakers

        

          
        

        

          SKODE is the compact ASCII shorthand. Skred is the runtime shell, native or in the browser. Pulp is the C audio engine underneath it: voices, buffers, tables, filters, modulators, samples. The speakers are where all the architecture stops being abstract. The more poetic version is: text becomes state, and state becomes air pressure.

Pulp voice engine

        
One voice is a signal path.

        

          
        

        

          This is the stage-friendly version of one Pulp voice from the voice structure diagram in learn.html. Pulp runs many voices; each voice is a small signal path with places to interfere. The full version includes more details, but the mental model is enough here: a waveform table is read at a frequency, shaped by filters and envelopes, then panned to output. Other voices can interfere through FM, AM, pan modulation, phase distortion, and filter modulation.

isolation and safety

            
Small parts are easier to restart.

          

          
        

        

          The safety strategy is architectural. Keep the audio engine small and sharp, then isolate it behind pipes, network ports, browser sandboxes, and process boundaries. Ports reduce the blast radius: easy to start, easy to kill, easy to restart. If Skred dies from a bad command, the host can kill it and spin it back up. The point is not that crashes are good; the point is that the blast radius is contained and recovery is cheap.

tiny protocol

        
Short commands change live state.

        

          
CUEskode stays where you leave it

          v0 w0 f220 a0.8
v1 m1 f6
v0 F1,40
        

        

          
          
          
          
        

        

          A short command changes the current engine state, and the next short command can build on it. That statefulness is what makes tiny command fragments useful live: you do not have to resend a whole patch every time. The compact technical framing is: stateful execution, stateless-looking text.
        

      


      

        

          

            
k-synth

            
Arrays can become sound.

          

          
        

        

          To populate an audio engine, you need data structures. Writing loops in raw C for lookup tables is tedious. k-synth lets me generate useful sound material from compact array expressions, then feed those tables into the same runtime. The original phrasing was: array code is a shortcut from math to timbre.
        

      


      

        

          

            
sk/dw8k-01.ks -> W10

            
The first DW-8000 wave is seven lines of text.

            

              
KSDW-8000 Wave 01: strings

              / DW-8000 Wave 01: strings
/ Perfect 1/h sawtooth. All 32 harmonics.
P: ~4096
H: !32
H: H+1
A: 1%H
W: w P $ A
            

            

              
              
            

          

          
        

        

          This is the bridge between k-synth and Skred. The source file is parts/sk/dw8k-01.ks in the repo, and that generated table corresponds to W10 in SKODE. It is modeled as a 1/h sawtooth with 32 harmonics, producing the first DW-8000-style string wave.

          The DW-8000 wave has its own slide now, so this one moves to k-synth drum generation. Drum samples can be generated from tiny k-synth patches. These patches are inspired by Roland 808-style drum voices: a swept sine kick, a detuned body plus noise snare, and a two-frequency cowbell cluster. They are text files that become runtime waveforms.
        

      


      

        
deferred execution

        
SKODE can schedule its own future.

        

          
TIMEworks from buttons, files, and the REPL

          v10l2 +.5 v11l2
~1 v12n69l1 ~1 v12n70l1 ~1 v12n69l1
        

        

          Deferred execution is not just a pattern feature. It works anywhere SKODE is parsed: from a button, from a file, or directly in the REPL. A command like +.5 waits a tempo-relative amount before running the next command. A command like ~1 schedules by a direct time-style delay.
        

      


      

        
patterns

        
Patterns make timed chunks reusable.

        

          
SEQstep pattern with timed events

          y0 %4
[v10l1 +.5 v10l0] x0
[v11l1] x2
Z1
        

        

          Patterns are step sequences. y0 selects pattern 0, %4 makes a four-step loop, bracketed SKODE is stored into a step with x0 or x2, and Z1 starts the pattern clock. Deferred commands inside a step let a single step contain timing, not just a single trigger.
        

      


      

        
909.sk

        
One file can build a groove.

        /w24,303
[sk/drums-snare.ks]/ks wait 250 k>d d>r /r300
[sk/drums-kick.ks]/ks wait 250 k>d d>r /r301
[sk/drums-chh.ks]/ks wait 250 k>d d>r /r302

y2 %4 [kick] yt
[+.5 v4 l1 +.25 v4l.5] x0
        

          
          
          
        

        

          This is the bigger patch-file version: a patch file can load samples, generate drums, define voices, and install patterns. Sample slots, generated drums, named chunks, voice setup, and pattern installation live in one text file. The demo point is that SKODE is not only a REPL trick; it can be saved, loaded, and reused.
        

      


      

        
live demo

        
External programs can talk SKODE to Skred.

        

          
          
          
          
        

        

          The live demo is not only typing commands into this deck. The useful boundary is that Skred accepts SKODE from outside programs too: a drum pad interface, a MIDI-to-SKODE translator, a multi-slider control GUI, and a looping app can all speak the same terse command surface.
        

      


      

        
what exists now?

        
The protocol is the instrument.

        

          What exists now: macros, patterns, generated drums, file-backed patches, and browser/native runtimes. The interesting part is not one synth. It is a small protocol surface that can sit between text files, generated tables, samples, runtimes, instruments, and humans. That is why the protocol itself starts to feel like the instrument.
